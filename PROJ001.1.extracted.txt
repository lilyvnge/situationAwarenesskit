4. n8n Workflow Pipelines (The Core Orchestration)
We define several n8n workflows, some triggered by webhooks/
cron
, others sub-workflows.
4.1 OSINT Ingestion Workflow
Trigger: Cron (every 5 min), webhooks, RSS poll, Telegram bot listener.Steps:
Fetch/Receive raw post from source (HTTP Request, Telegram Trigger, etc.).
Deduplication: Compute hash (MD5) of 
source_url
 + text. Check against Redis cache or DB. If duplicate, stop.
Sanitization Node: Call Presidio API to redact PII (optional). Use regex to strip control chars.
Source Credibility: Look up source domain/handle in a Config node (maintained table) and attach 
source_rating
 (A-F) and 
source_tier
.
Language Detect & Translate: n8n HTTP node to 
LibreTranslate
 API; store both original and English text.
Geoparse
: Use a self-hosted 
spaCy
 API or a geoparsing node that extracts locations and coordinates (via 
Nominatim
/
Geonames
). If no coordinates found, fallback to country centroid based on mention.
Embedding Generation: Send 
clean_english_text
 to an embedding service, receive vector.
Store OSINT Observation: Insert into 
osint_sources
 table via Postgres node.
Enqueue for Classification: Publish message to Redis channel 
pending_classification
 with 
post_id
.
4.2 AI Classification Workflow
Trigger: Redis pub/sub listener or an n8n polling node that checks a pending status column.Steps:
Fetch post by ID from DB.
Call AI Model: HTTP request to the 
vLLM
/
Ollama
 server. The prompt includes the event taxonomy and requires structured JSON output.
Validate JSON Schema: If invalid, retry or flag for human review.
Extract Entities: From the JSON, get 
event_class
, actor, target, location, etc.
Match/Resolve Actors: Query actors table via fuzzy match (or exact). If new actor, create a placeholder and flag.
Insert Event: into events table with geometry from coordinates, AI confidence score, and FK to OSINT source.
Auto-linkage Trigger: Call a webhook that runs a mini linkage check for the new event (next workflow).
Update Status: Mark OSINT post as classified.
4.3 Automated Linkage Workflow (Scheduled or On-Event)
Trigger: On new event or nightly 
cron
.Steps:
Spatial-temporal cluster: Query events within last 24h, using 
ST_DWithin
 and time difference < 12h, same country.
Actor similarity: For each new event, find events in past 90 days where 
actor_initiator_id
 or 
actor_target_id
 matches exactly.
Vector similarity: Use 
pgvector
 to find events with description embedding cosine similarity > 0.85, different from itself.
Heuristic scoring: Combine spatial, temporal, actor, and semantic similarity into a 
link_confidence
 (0.0–1.0).
Propose Links: Insert candidate links into 
event_links
 with 
link_confidence
 and status proposed; notify analyst queue if above threshold.
Analyst review: An n8n workflow can email/Slack an analyst to confirm or reject.
4.4 Guardrails & Compliance Workflows
Sensitive Keyword Alert: If text contains specific weapon system codes or classified codewords (configurable list), route to a secure vault, not the main dashboard (requires separate handling).
Quarantine: Low source rating (E-F) posts can be stored but not auto-linked; held for human analysis.

6. API Layer (
FastAPI
 Skeleton)
Endpoints needed for the dashboard:
GET /events – paginated list with filters (class, country, date range, actor, confidence). Returns 
GeoJSON
 
FeatureCollection
.
GET /events/:id – single event with all linked OSINT, actors, proposed links.
POST /events/:id/links – analyst creates/confirms a link.
GET /events/network – returns graph data (nodes and edges) for linkage visualization (
event_ids
, relationship).
GET /stats – aggregated counts, timeline data for charts.
POST /analyst/review – approve/discard/classify event.
WebSocket /
ws
/events – pushes new confirmed events in real-time (via Redis pub/sub).
The AI classification service can be exposed as a microservice with a single endpoint /classify that takes text and returns structured JSON, so n8n can call it directly.
7. Frontend Dashboard Components
Map View: Uses 
MapLibre
 GL JS with custom vector tiles (or just 
GeoJSON
 layers from API). Layers: event clusters, heatmap, filtered by class (icons for military, political, etc.). Clicking an event opens a sidebar.
Timeline & Analytics: A timeline chart showing event counts over time, filterable by region. A table view with sorting.
Network Graph Panel: When an event is selected, show its links as a force-directed graph (react-force-graph). Allows analysts to drag, explore, and click to navigate.
Analyst Workbench: A queue of pending events (low confidence, proposed links). A form to edit classification, confirm actors, create links manually.
Watchlist & Alerts: Users can set geographic/country/actor watchlists and receive in-app notifications.
Authentication via JWT; role-based access (viewer, analyst, admin).

Backup: DB backups daily, stored encrypted. n8n workflows exported to Git.

10. Development Roadmap (Phases)
Phase 1 – MVP (2–3 months)
Set up n8n, PostgreSQL, basic ingestion from 1-2 sources (e.g., RSS + Telegram).
Simple classification using pre-trained LLM (no fine-tuning) with basic event taxonomy.
Minimal dashboard: map with event points from DB.
Manual event creation/editing by analysts.
Phase 2 – Analyst Features (2 months)
AI fine-tuning on military/geopolitical event data (using CAMEO or custom dataset).
Entity (actor) extraction and actor table population.
Automated linkage suggestions (spatial + actor).
Analyst review queue, link confirmation UI, network graph.
Phase 3 – Scale & Enrich (2 months)
Advanced guardrails, PII redaction, disinformation filters.
I&W pattern recognition (complex event processing).
Multi-tenancy if needed.
Mobile-friendly analyst notifications.
This setup gives you a production-ready foundation that stays true to the intelligence analyst’s needs while being built with a modern, modular stack.
